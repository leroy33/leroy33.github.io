初探重构

关于strtod()函数

函数会舍弃任何空白符(由std::isspace()确定)，直至找到首个非空白符。然后它会取用尽可能多的字符，以构成合法的浮点数表示，并将它们转换成浮点值。函数设置str_end所指向的指针指向最后被转译字符的后一字符，若str_end为空指针，则忽略它。若无法进行转换，则返回 0 并将*str_end设为str。

所以只有该字符串中第一个数字前面只有空格(不能有其他字符)，才能将第一个合法的字符串读出来。然后str_end指向这个合法数字的下一字符。

有一些JSON库会选择更复杂的方法(而不是使用strtod())，如支持 64 位带符号／无符号整数，自行实现转换。就针对另一JSON库——RapidJSON，解析／生成数字类型可以说是其中最难实现的部分，也是它高效性能的原因。

总结

这一章重新回顾了TTD的一些流程，加深了我对单元检测的印象。本章主要以JSON文本的数字语法及解析展开，让我见识到了怎样把如此的JSON语法写成代码。因为是处理数字，所以很多地方涉及到浮点型数据类型实现的问题，可以查看其他讲计算机系统或操作系统的书籍、资料。

另外我加深了对这个框架的理解。首先这里面定义了很多宏，都是解决代码重复的问题的。然后有一JSON数据结构，这一章开始逐渐“壮大”(新添了double n)。解析器主要分几块，首先是根据JSON-text = ws value ws规则得到value，然后解析value，然后对每一种JSON数据类型都有分模块处理。如果解析成功，那么保存对应值，不成功就返回错误类型。对于解析器还有一写接口，来获取解析得到的值。

然后是测试部分代码，有几个变量是来计算总测试次数和成功测试次数的，然后测试有个总入口test_parse()，所有的测试模块都包含在内，这些测试模块包括：测试获取值是否正确的模块，以及测试返回错误类型是否正确的模块。他们主要都是使用EXPECT_EQ_BASE来判断实际的返回值和预期的返回值是否相等，并更新测试计数变量(从这个BASE也可以看出其他的模块都是基于此)。
