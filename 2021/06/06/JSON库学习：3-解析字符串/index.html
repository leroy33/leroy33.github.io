JSON 字符串语法

JSON字符串，使用双引号括起来，且有反斜杠\作为转义符，和C语言挺像的。

quotation-mark指的是引号，escape指的是转义的字符(有9种)，unescaped指的是未转义的字符。最后一种转义字符\uXXXX(\u表示后面XXXX是一个Unicode字符)较特殊，本章暂不处理。

字符串表示

与C语言不同，C语言中遇到结尾字符\0即可知道字符串结束了，但JSON中允许有该字符，如Hello\u0000World是单个字符串，且字符长度为11(\u0000是一个Unicode字符)。因此在JSON节点的结构体中加入一个变量记录字符串长度。

缓冲区与堆栈

在解析字符串时，由于不知道字符串的具体长度，因此需要把解析结果先放于临时缓冲区中(lept_context)，再用lept_set_string()把lept_context中的结果设值(解析数组和对象也是如此策略)。此时使用了动态数组(dynamic array)这一结构，这与std::vector的策略是类似的(可参考《Primer C++ 5th》P317)。

解析字符串(或数组和对象)时，我们都可以以先进后出的方式访问这个动态数组，这正好和栈这一数据结构吻合。因此把这一动态堆栈添加在lept_context中。

size是目前堆栈容量，top是栈顶(栈顶指向是与最后放入的元素相关的)。在创建lept_context时要初始化stack，用完之后要释放。

然后实现该堆栈的入栈和出栈。注意这个栈的最小单元是字节(对于数组或对象，一个元素可能不止1字节)，size可以控制字节数。

注意当压入的空间不足时，是将原容量扩展为1.5倍的。还要注意void*类型返回值并不是指无返回值，而是指返回的一个指针，这个指针无类型指定(可将空类型指针强制转为任意类型指针)。

解释一下这俩函数，首先是压入函数，输入需要缓冲的变量及开辟空间大小(单位为比特)，返回值是堆栈中应该放入内容的地址。其次是弹出函数，输入参数和前者是一样的，返回值是弹出后的栈顶地址。

解析字符串

解析字符串的过程是，先备份栈顶(方便计算长度)，再解析字符并压入堆栈，再根据备份的栈顶计算字符串长度并全部弹出，最后设置到值里。注意下面暂时没有处理转义字符和一些不合法的字符校验。

总结和练习

之前的单元都是固定长度的数据类型（fixed length data type），而字符串类型是可变长度的数据类型（variable length data type），因此本单元花了较多篇幅讲述内存管理和数据结构的设计和实现。字符串的解析相对数字简单，以下的习题难度不高，同学们应该可轻松完成。

  1. 编写lept_get_boolean()等访问函数的单元测试，然后实现。
  2. 实现除了\u以外的转义序列解析，令test_parse_string()中所有测试通过。
  3. 解决test_parse_invalid_string_escape()和test_parse_invalid_string_char()中的失败测试。
  4. 思考如何优化lept_parse_string()的性能，那些优化方法有没有缺点。

性能优化的思考

  • 若整个字符串中没有转义字符，那么其实我们使用堆栈就是“多此一举”的，因为我们完全可以直接从JSON文本中复制字符串到JSON节点的字符串变量中。可能的优化方法是，先扫描一遍需要JSON字符串文本，知道遇见\0、\"和\\以及不合法的非转义字符，才使用现在的堆栈的方法，否则直接复制。
  • 对于扫描遇到没转义字符，可以考虑用SIMD加速。关于这个知识我也没搞懂，所以仅贴出教程。
  • 考虑在gcc/clang上使用__builtin_expect()来处理低概率事件。

总结

这一章是字符串的解析，JSON字符串语法和C语言语法比较类似，由于字符串长度不定，因此给JSON节点增加了一个可动态分配内存的结构。且中间可能遇到转义字符，所以给lept_context构造一个堆栈空间，且该空间的容量控制方法类似于std::vector。接下来就是解析过程，也就是根据语法将堆栈中的元素一个一个解析出来。

感觉经过这一章之后，慢慢熟悉了由单元测试驱动函数编写的过程。另外要注意的是，本章解析的字符串仅限ASCII，关于Unicode会在后面提到。
